시스템 개요

🎯 목적

iPhone LiDAR 센서로 골프 그린 지형을 3D 스캔한 후, 볼-홀 간 최적 퍼팅 궤적(브레이크 방향·속도)을 실시간으로 계산하여 AR 화면에 오버레이 표시

📦 주요 모듈

  • LiDARScanner — 깊이 데이터 수집
  • DepthImageRenderer — false-color 시각화
  • TerrainAnalyzer — 경사 분석
  • PuttingPhysics — 궤적 계산
  • TrajectoryOverlayView — 결과 렌더링

⚙️ 그리드 설정

  • 해상도: 300 × 300 셀
  • 셀 크기: 5 cm
  • 커버리지: 15m × 15m
  • 유효 거리: 0.1 ~ 5.0 m
  • 픽셀 처리 step: 4px 간격

이미지 기반 거리측정 원리

📷 ARKit SceneDepth 깊이 맵

iPhone Pro LiDAR는 적외선 레이저를 방사하고 ToF(Time-of-Flight) 방식으로 각 픽셀의 깊이(d)를 kCVPixelFormatType_DepthFloat32 포맷으로 반환합니다. ARKit은 두 가지 깊이 맵을 제공합니다:


맵 종류특징사용
sceneDepth원시 LiDAR 데이터, 노이즈 포함신뢰도 맵과 함께 원본 제공
smoothedSceneDepthARKit 내부 시계열 평활화 적용✅ 이 앱에서 우선 사용
guard let depthData = frame.smoothedSceneDepth ?? frame.sceneDepth // smoothed depth 우선, 없으면 raw depth 사용 (LiDARScanner.swift:282)

🔬 신뢰도 맵 (ConfidenceMap)

각 픽셀마다 ARKit이 신뢰도 레벨을 UInt8로 제공:


값레벨가중치
0Low0.3
1Medium0.6
2High1.0
let confBase = confidence == 2 ? 1.0 : confidence == 1 ? 0.6 : 0.3

ℹ️
픽셀 처리 방식전체 깊이 맵을 step=4 간격으로 순회합니다. 깊이 맵 해상도가 약 256×192라면 프레임당 약 3,072개의 포인트를 처리합니다. step=2로 줄이면 4배 더 많은 포인트(밀도 향상)를 얻을 수 있으나 처리 시간이 증가합니다.

좌표 변환 파이프라인

Fig.2 — 깊이 픽셀 (u,v,z) → 카메라 좌표계 → 월드 좌표계 → 그리드 인덱스 변환의 3단계 파이프라인

📐 Step 1: 픽셀 → 카메라 좌표계

카메라 내부 파라미터(intrinsics)를 사용하여 깊이 픽셀 (u, v)를 3D 카메라 좌표로 역투영합니다:

let intrinsics = frame.camera.intrinsics let fx = intrinsics[0][0] // X축 초점 거리 (픽셀 단위) let fy = intrinsics[1][1] // Y축 초점 거리 let cx = intrinsics[2][0] // X축 주점 (principal point) let cy = intrinsics[2][1] // Y축 주점 // 역투영 공식 Xc = (u - cx) × z / fx // 카메라 X (좌우) Yc = (v - cy) × z / fy // 카메라 Y (상하) Zc = -z // 카메라 Z (깊이, ARKit은 -Z가 전방) localPt = SIMD4(Xc, Yc, -z, 1.0) // 동차 좌표

🌍 Step 2: 카메라 → 월드 좌표계

4×4 카메라 변환 행렬(T_cam)로 월드 좌표로 변환합니다. ARKit 월드계에서 Y축이 위(up)입니다:

worldPt = cameraTransform × localPt // cameraTransform = frame.camera.transform (4×4 SE(3) 행렬: 회전 R + 이동 t) // worldPt.x = 월드 X, worldPt.y = 월드 높이(Y=up), worldPt.z = 월드 Z

🗺️ Step 3: 월드 → 그리드 인덱스

그리드 원점(gridOriginX, gridOriginZ)을 기준으로 상대 좌표로 변환 후 셀 인덱스를 계산합니다:

relX = worldPt.x - gridOriginX relZ = worldPt.z - gridOriginZ gridRange = 300 × 0.05 / 2 = 7.5 m // 그리드 반폭 gx = Int((relX + gridRange) / cellSize) // 0 ~ 299 gz = Int((relZ + gridRange) / cellSize) // 0 ~ 299 // 그리드에 가중치 누적 accumulatedHeights[gz][gx] += worldPt.y × confWeight accumulatedConfidence[gz][gx] += confWeight accumulatedCounts[gz][gx] += 1

기준점 및 기준평면 설정

📍 그리드 원점 설정

첫 번째 유효 프레임에서 카메라 전방 7.5m 지점을 그리드 중심으로 설정합니다. 이렇게 하면 그리드가 카메라 기준 전방 0~15m 범위를 커버합니다:

camPos = frame.camera.transform.columns.3 camFwd = -frame.camera.transform.columns.2 // 카메라 전방벡터 (ARKit -Z) halfRange = 300 × 0.05 / 2 = 7.5 m gridOriginX = camPos.x + (fwdX / |fwd|) × halfRange gridOriginZ = camPos.z + (fwdZ / |fwd|) × halfRange

📏 높이(Y) 기준평면

ARKit 월드 좌표계의 Y축이 절대 높이 기준입니다. ARKit의 초기화 시 지면 평면 감지(planeDetection: horizontal)로 Y=0 기준면을 설정합니다. 각 셀의 높이는 worldPt.y 값의 가중 평균으로 결정됩니다.

h_cell = Σ(worldY × confWeight) / Σ(confWeight) // 신뢰도 가중 평균 → 높은 신뢰도 측정값을 더 많이 반영

📐 카메라 각도 기준 (tiltQuality)

카메라가 완전히 수평이면 그린 표면을 정확히 측정할 수 없습니다. 카메라 forward 벡터의 Y 성분으로 "아래를 향하는 정도"를 계산하고, 이를 신뢰도 가중치에 곱합니다:

fwdY = -cam.columns.2.y // 카메라가 아래를 향할수록 양수 (0~1) pointingDownScore = clamp(fwdY, -1, 1) // 0.15 미만: 거의 수평 → 해당 프레임 유효 포인트로 처리 안함 guard pointingDownScore > 0.15 else { return } confWeight = confBase × pointingDownScore // 기울기 품질 반영
⚠️
목표 각도: 30° pointingDownScore = sin(30°) ≈ 0.5 — 카메라를 30° 아래로 향할 때 가장 균형 잡힌 커버리지와 측정 정밀도를 얻습니다. tiltScore는 이 목표에서의 편차를 점수화합니다.

핵심 계산 수식

🔢 역투영 (Back-Projection)

// 핀홀 카메라 모델 역투영 Xc = (u - cx) · z / fx Yc = (v - cy) · z / fy Zc = -z // ARKit: 카메라 전방 = -Z // 동차 좌표 (Homogeneous) P_cam = [Xc, Yc, Zc, 1]ᵀ // 월드 변환 P_world = T_cam · P_cam

📊 경사도 계산 (Central Difference)

// 기본 중앙차분 (TerrainAnalyzer.calculateSlope) slope_x = -(h[x+1,y] - h[x-1,y]) / (2 · cellSize) slope_y = -(h[x,y+1] - h[x,y-1]) / (2 · cellSize) // 고정밀 다중스케일 가중 중앙차분 (r=3 셀 반경) for dr in 1...3: w = 1/dr // 거리 반비례 가중치 scale = 2 · dr · cellSize gx += w · (h[x+dr,y] - h[x-dr,y]) / scale gy += w · (h[x,y+dr] - h[x,y-dr]) / scale wSum += w slope = Vector2(-gx/wSum, -gy/wSum) slope_angle_deg = atan(|slope|) × 180/π

🏌️ 퍼팅 브레이크 & 속도 계산

// 경사 영향 계수 (slopeEffect) slopeEffect = 1.0 - resistancePercent / 100.0 // 저항 0%=경사 최대반영, 100%=직선 퍼팅 // 브레이크 양 (Quadratic Bezier 컨트롤 포인트 오프셋) breakAmount = avgCrossSlope × distance × slopeEffect / 4.0 // 컨트롤 포인트 mid = (ballPos + holePos) × 0.5 controlPt = mid - perpDir × (2.0 × breakAmount) // Bezier 궤적 공식 (t ∈ [0,1]) B(t) = (1-t)² × P_ball + 2(1-t)t × P_control + t² × P_hole // 필요 속도 (운동-마찰 에너지 보존) effectiveFriction = 0.065 × (1 + resistancePercent/100) speed = √(2 × μ × g × distance) × 1.025 // g=9.81 m/s², 1.025: 홀 통과 여유 (102.5% 힘)

🌊 가우시안 스무딩 커널

// 9×9 커널, σ = 9/3 = 3 G(dx, dy) = exp(-(dx² + dy²) / (2σ²)) // 신뢰도 가중 적용 smoothed[x,y] = Σ(G(dx,dy) · conf[x+dx,y+dy] · h[x+dx,y+dy]) / Σ(G(dx,dy) · conf[x+dx,y+dy])
Fig.3 — 가중 평균 → 빈 셀 보간 → 미디언 필터(3×3) → 가우시안 스무딩(9×9) → 최종 높이 맵 처리 파이프라인

✅ 현재 보정 방식 — ARKit 절대 좌표계 활용

이 앱은 높이 편차를 명시적으로 보정하지 않아도 되는 구조로 설계되어 있습니다. 그 이유는:


  1. ARKit 월드 Y좌표 사용: worldPt.y는 ARKit이 추적한 절대 세계 Y좌표입니다. 카메라가 1.4m에 있든 1.8m에 있든, 그린 표면의 worldPt.y는 동일하게 계산됩니다.
  2. 카메라 transform 역변환: T_cam은 카메라의 위치와 방향을 포함하므로, 높이가 달라도 동일한 그린 점은 동일한 worldPt.y로 수렴합니다.
  3. 다중 프레임 가중 누적: 여러 프레임에서 신뢰도 가중 평균을 내므로 단일 높이 오차의 영향이 희석됩니다.

⚠️
현재 누락된 높이-시차 보정 카메라 높이가 1.4m → 1.8m로 변하면 같은 30° 각도에서 그린 표면의 측정 거리가 달라집니다. 높이 h에서 30° 각도로 바라본 그린 교점까지의 거리 = h / sin(30°) = 2h. 따라서 1.4m에서는 2.8m, 1.8m에서는 3.6m 거리의 점을 같은 픽셀 위치에서 바라봅니다. 이 차이는 약 0.8m 거리 오차를 유발할 수 있습니다.

// 권장: CoreMotion 고도계로 카메라 높이 추정 후 스케일 보정 cameraHeight_m = -motionAltitude × 상수 // barometer 기반 // 또는: ARKit planeDetection으로 감지된 지면과 카메라 Y 간 거리 groundY = detectedHorizontalPlane.transform.columns.3.y cameraHeight = cameraTransform.columns.3.y - groundY // 높이에 따른 유효 거리 범위 동적 조정 maxValidDepth = min(5.0, cameraHeight / sin(tiltAngle_rad) + 1.0)

🎯 신뢰도 × 자이로 복합 가중치

confWeight = confBase × tiltQuality // confBase: ARKit 신뢰도 (0.3/0.6/1.0) // tiltQuality: 카메라 기울기 점수 (0.15~1.0) // 최소 가중치: 0.3×0.15=0.045 // 최대 가중치: 1.0×1.0=1.0

tiltQuality가 낮은 (수평에 가까운) 프레임은 신뢰도에 관계없이 낮은 가중치를 받아 최종 높이 맵 영향이 감소합니다.

🔍 빈 셀 보간 알고리즘

for iteration in 0..<5: for each empty cell (conf < 0.01): neighborSum = 0, weightSum = 0 for (dx,dy) in -2...2 × -2...2: d = sqrt(dx²+dy²) w = conf[nx,ny] / d neighborSum += height[nx,ny] × w weightSum += w if weightSum > 0: height[x,y] = neighborSum / weightSum conf[x,y] = 0.3 // 보간된 셀은 낮은 신뢰도

품질 지표 (MeasurementQuality)

📊 5개 품질 지표

지표목표값계산 방법
averageConfidence> 0.6전체 픽셀 신뢰도 평균 (step=8 샘플)
coveragePercent> 0.7채워진 셀 / 바운딩박스 내 셀 수
stabilityScore> 0.51 - avg_movement × 50 (최근 30 pose)
lightingScore> 0.5ambientIntensity / 1000.0
tiltScore> 0.21 - |fwdY - 0.5| / 0.5 (목표: sin30°=0.5)

🤖 자동 종료 조건

isGood = coveragePercent > 0.70 && averageConfidence > 0.60 && stabilityScore > 0.50 && tiltScore > 0.20 // 연속 3 프레임 isGood == true → onAutoStopReady 발동 if ++autoQualityFrames >= 3: onAutoStopReady()

또한 3초 동안 커버리지 변화가 0.2% 미만이면 스캔 정체로 판단하여 자동 완료합니다.

[우선순위 높음] 3D 인터랙티브 메쉬그리드 (Web/Metal)

현재는 2D 투영 오버레이만 지원합니다. SceneKit 또는 Metal을 이용한 실시간 3D 메쉬 렌더링으로 사용자가 그린을 직접 회전·확대해서 볼 수 있도록 개선하면 경사 이해도가 크게 향상됩니다.

// Metal Shader: 높이 → 색상 매핑 float4 heightColor(float h, float minH, float maxH) { float t = (h - minH) / (maxH - minH); return mix(lowColor, highColor, t); }
높음 Metal / SceneKit
📏

[우선순위 높음] 적응형 그리드 해상도 (LOD)

현재 300×300 고정 해상도는 가까운 영역(볼 주변)과 먼 영역(5m 이상)을 동일하게 처리합니다. 볼-홀 주변을 고해상도(2.5cm), 외곽을 저해상도(10cm)로 처리하는 Level-of-Detail 그리드로 개선하면 정밀도와 성능을 동시에 향상시킬 수 있습니다.

높음 알고리즘 개선
🗺️

[우선순위 중간] 등고선 간격 개선 + 레이블링

현재 등고선은 10단계 고정 간격입니다. 실제 고도 차이에 기반한 절대 높이 기준 등고선(예: 1cm 간격)과 각 등고선에 높이값 레이블을 추가하면 훨씬 직관적인 맵이 됩니다.

// 개선: 절대 높이 기준 등고선 let contourInterval = 0.01 // 1cm 간격 let levels = stride(from: minH, through: maxH, by: contourInterval)
